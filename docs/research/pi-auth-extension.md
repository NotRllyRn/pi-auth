# Research: Named authentication profiles for pi

## Summary

Pi 0.80.6 has a flat, provider-keyed credential store: one persisted credential per provider. A normal extension command or input handler **cannot replace or observe built-in `/login`**, and there is no auth-change event before `AuthStorage.login()` writes its result. A pure extension still has three possible seams: own a different login command; register one provider alias per named profile; or experimentally wrap each registered OAuth provider's public `login()` method so the built-in flow passes through extension code before `AuthStorage.login()` persists the returned credential. Fully native named profiles—especially canonical provider identity, API-key parity, transactional profile updates, and first-class `/login`/`/logout` integration—requires a small upstream core/API change.

> Version examined: installed `@earendil-works/pi-coding-agent@0.80.6` and its nested `@earendil-works/pi-ai@0.80.6`; package metadata points to the official `earendil-works/pi` repository. Precise installed paths and symbols are cited because they exactly match the runtime under design.

## Findings

1. **Built-in `/login` is handled in the interactive editor before extension command/input processing.** `InteractiveMode.setupEditorSubmitHandler()` recognizes `/login` and `/login <provider>`, clears the editor, and calls `handleLoginCommand()`; only unhandled input later reaches `AgentSession.prompt()`. `AgentSession.prompt()` checks extension commands and then emits `input`, but that path is never reached for a built-in login submitted in the TUI. Extension commands conflicting with built-ins are omitted from autocomplete and diagnosed; multiple extension-command collisions receive `:1`, `:2` suffixes, but the interactive built-in still owns exact `/login`. Therefore an extension cannot intercept or override built-in `/login` through `registerCommand()` or `input`. [Official source, command-conflict logic](https://github.com/earendil-works/pi/blob/9b62f1f87c3429dc29bf7c33bef082d4be13c8a1/packages/coding-agent/src/modes/interactive/interactive-mode.ts#L330-L363); installed symbols: `dist/modes/interactive/interactive-mode.js` → `setupEditorSubmitHandler`, `getBuiltInCommandConflictDiagnostics`; `dist/core/agent-session.js` → `prompt`, `_tryExecuteExtensionCommand`; `dist/core/extensions/runner.js` → `resolveRegisteredCommands`, `getCommand`.

2. **The built-in OAuth result goes straight from the provider to a provider-keyed overwrite, with no extension event.** `InteractiveMode.showLoginDialog()` calls `modelRegistry.authStorage.login(providerId, callbacks)`. `AuthStorage.login()` awaits `provider.login(callbacks)` and immediately calls `set(providerId, {type: "oauth", ...credentials})`. The extension event union contains model/session/input/provider-request events but no auth/login event, and there is no callback around `AuthStorage.set()`. A failed/cancelled login leaves the previous credential intact because `set()` occurs only after `provider.login()` resolves; a successful login replaces that provider’s old credential. A command/input hook therefore cannot capture the result. There is, however, an experimental public-API seam worth prototyping: during `session_start`, read the registered providers through `ctx.modelRegistry.authStorage.getOAuthProviders()`, retain each original provider object, and call `pi.registerProvider(provider.id, { oauth: wrappedProvider })`. The wrapper can snapshot the old canonical credential, delegate to `original.login(callbacks)`, and use `callbacks.onSelect()` after token validation but before returning the new credential to `AuthStorage.login()`. Risks: no transaction spans the extension sidecar write and the subsequent core write, wrapper registration/reload ordering must be proven, and this does not intercept built-in API-key login. Installed sources: `dist/modes/interactive/interactive-mode.js` → `showLoginDialog`; `dist/core/auth-storage.js` → `AuthStorage.login`; `dist/core/model-registry.js` → `registerProvider`, `applyProviderConfig`; `dist/core/extensions/types.d.ts` → `ExtensionEvent`; nested `@earendil-works/pi-ai/dist/utils/oauth/types.d.ts` → `OAuthProviderInterface.login`. [Official login-flow history](https://github.com/badlogic/pi-mono/commit/010e9acfe959f437613bcba7139b264012ca43a4).

3. **`auth.json` is deliberately flat and allows exactly one credential value per string key.** The schema is `AuthStorageData = Record<string, AuthCredential>`, where a credential is `{type:"api_key", key, env?}` or `{type:"oauth", refresh, access, expires, ...providerFields}`. `get`, `set`, `remove`, `list`, `has`, and `getAll` all operate on that single top-level key. There is no profile/name/active-pointer field. Installed declarations: `dist/core/auth-storage.d.ts` → `ApiKeyCredential`, `OAuthCredential`, `AuthStorageData`, `AuthStorage`. [Official source](https://github.com/earendil-works/pi/blob/9b62f1f87c3429dc29bf7c33bef082d4be13c8a1/packages/coding-agent/src/core/auth-storage.ts).

4. **Writes are locked and merge the latest disk state, but are not crash-atomic.** `FileAuthStorageBackend` creates the parent/file with modes `0700`/`0600`. Synchronous operations retry `ELOCKED` ten times; async refresh uses `proper-lockfile` retries, `stale: 30000`, and a compromise callback. `persistProviderChange()` reparses current disk content while holding the lock, merges only the requested provider, then serializes the full object. This prevents cooperating processes updating different providers from blindly losing one another’s writes; same-provider writes are last-writer-wins. However the final operation is in-place `writeFileSync(authPath, next)`, not temp-file + `fsync` + atomic rename, so process/host failure can leave truncated JSON. Parse/load failure is retained as `loadError`; mutating operations retry reload and refuse to write if loading still fails. Installed source: `dist/core/auth-storage.js` → `AUTH_FILE_WRITE_OPTIONS`, `FileAuthStorageBackend.withLock`, `withLockAsync`, `AuthStorage.persistProviderChange`.

5. **OAuth refresh is demand-driven, provider-keyed, serialized, and persisted.** Credential resolution priority is runtime override, stored API key, stored OAuth, then environment. For expired OAuth, `getApiKey()` calls `refreshOAuthTokenWithLock()`, which reloads current disk state under an async lock, rechecks expiry, refreshes, and writes the new credential into the same provider key. On failure it reloads to see whether another process succeeded; otherwise it returns `undefined` but preserves the old credential for reauthentication/retry. Non-expired credentials are converted with `provider.getApiKey()`. Installed source: `dist/core/auth-storage.js` → `AuthStorage.getApiKey`, `refreshOAuthTokenWithLock`. The underlying `getOAuthApiKey(providerId, credentials)` indexes exactly `credentials[providerId]` and invokes that registered provider’s `refreshToken()`. Installed source: nested `@earendil-works/pi-ai/dist/utils/oauth/index.js` → `getOAuthApiKey`. This means an inactive credential copied only to an extension sidecar will never be refreshed by core; an independently keyed provider alias remains refreshable whenever its alias is resolved.

6. **Model credential lookup is based solely on `model.provider`.** `ModelRegistry.getAvailable()` uses the non-refreshing `hasConfiguredAuth()`. Actual request setup calls `getApiKeyAndHeaders(model)`, which asks `authStorage.getApiKey(model.provider, {includeFallback:false})`, then considers provider config. `isUsingOAuth()` likewise reads the credential at `model.provider`. `refresh()` rebuilds registered API/OAuth providers and model lists, and applies an OAuth provider’s `modifyModels()` only using the credential under that provider ID. Installed source: `dist/core/model-registry.js` → `getAvailable`, `hasConfiguredAuth`, `getApiKeyAndHeaders`, `isUsingOAuth`, `refresh`, `loadModels`; declarations in `dist/core/model-registry.d.ts`. [Official source](https://github.com/earendil-works/pi/blob/9b62f1f87c3429dc29bf7c33bef082d4be13c8a1/packages/coding-agent/src/core/model-registry.ts).

7. **The extension API exposes enough UI/model seams to build an independent profile manager.** Relevant APIs are: `registerCommand`; `ctx.modelRegistry` and its public `authStorage`; `pi.registerProvider()`/`unregisterProvider()` for dynamic provider/model/OAuth registration; `pi.setModel()` plus `model_select`; `ctx.ui.setStatus()` for a compact persistent active-name label; `setFooter()` if replacing the entire footer is acceptable; `select`/`input`/`confirm`; and `ctx.ui.custom(..., {overlay:true})` for a richer popup selector. Custom overlays are TUI-only; RPC returns defaults/no custom component, and print/JSON have no UI. Installed docs: `docs/extensions.md` → “ExtensionAPI Methods”, “pi.registerCommand”, “pi.setModel”, “pi.registerProvider”, “Widgets, Status, and Footer”, “Custom Components / Overlay Mode”, “Mode Behavior”. Examples: `examples/extensions/model-status.ts`, `custom-footer.ts`. [Official extension docs](https://github.com/earendil-works/pi/blob/9b62f1f87c3429dc29bf7c33bef082d4be13c8a1/packages/coding-agent/docs/extensions.md).

8. **Best pure-extension design: a distinct `/auth-profile` command plus one provider alias per profile.** For a canonical provider `P` and named profile `N`, register alias `P::N` with cloned models and a cloned/wrapped OAuth provider; store the credential under the alias key. Because model requests resolve `model.provider`, selecting the cloned alias model makes core refresh and persist that profile independently without rewriting `P`. The popup can add/login, activate, rename, and delete aliases; `setStatus("auth-profile", "auth: N")` displays the active name. On activation call `pi.setModel(aliasModel)`; on deletion, unregister the alias and remove its credential. This uses supported public APIs and avoids a sidecar containing live OAuth credentials. Caveats: aliases appear as distinct providers/models, cloned `modifyModels()` may need a wrapper because built-ins may filter by canonical provider ID, provider display/login menus may expose aliases, model/session history records the alias rather than canonical provider, and inactive tokens refresh only when explicitly resolved (activation or a deliberate maintenance call to `authStorage.getApiKey(alias)`). Installed sources: `dist/core/model-registry.js` → `registerProvider`, `applyProviderConfig`; `dist/core/auth-storage.js` → `getApiKey`; `docs/extensions.md` → `registerProvider` and `setModel`.

9. **Fallback pure-extension design: sidecar profiles plus canonical-slot swapping, but it is less safe.** The extension can own the OAuth flow by calling the provider’s public `login(callbacks)` itself, retain the returned credential in a `0600` sidecar, and copy the selected credential into `authStorage.set(P, credential)`. It can refresh sidecar entries by calling `provider.refreshToken()` and persisting the result. This preserves canonical models, but two pi processes choosing different profiles race on the same canonical key; requests already in flight and model-specific `modifyModels()` state can mismatch; and core refresh updates only the active copy, so the extension must reconcile refreshed data back into the sidecar. File watching built-in `auth.json` after `/login` is not an acceptable seam: it is post-overwrite, has no completion/provenance event, and introduces races.

10. **Native semantics require upstream support.** The clean core change is a backward-compatible schema/API such as `{version, providers:{P:{active:"name", profiles:{name:credential}}}}`, or an internal credential-ID abstraction while accepting legacy flat entries. `AuthStorage` should expose list/add/rename/remove/select and `login(provider, profile, callbacks)`; request resolution should return and refresh the selected profile under one lock; model registry should continue seeing canonical `model.provider`; and interactive `/login`/`/logout` should call these APIs and emit auth-change/login-result events. Also allow extension command override/interception only through an explicit, security-conscious API rather than relying on editor dispatch. Without this, a pure extension can provide the requested workflow only with alias-visible semantics or risky slot swapping.

11. **Security, migration, concurrency, and test requirements are substantial.** Keep refresh/access tokens out of session entries, status text, logs, autocomplete metadata, and provider config diagnostics; use `0600` files and `0700` dirs; never put secrets in project-local state; redact errors; and consider OS keychain/encryption at rest. Migrate by importing each legacy top-level provider as a named `default` profile without deleting the original until a validated atomic write succeeds. Use a versioned schema and reject duplicate/empty/path-like names. All add/rename/delete/select/refresh operations need one cross-process lock and atomic temp-write/rename; refresh must compare the credential generation or refresh token before commit so a stale refresh cannot overwrite a newer login. Test: legacy migration/rollback, permissions, malformed/truncated files, rename/delete active profile, duplicate names, failed/cancelled login preservation, refresh rotation for active and inactive profiles, two-process select/refresh/login races, lock compromise/staleness, provider-specific extra OAuth fields and `modifyModels`, alias registration/reload, model switch failure rollback, no-UI modes, and secret-redaction snapshots. Existing official tests are the right pattern for backend locking and refresh concurrency. [Official auth-storage tests](https://github.com/badlogic/pi-mono/blob/38f18be4/packages/coding-agent/test/auth-storage.test.ts); [reported lock contention history](https://github.com/badlogic/pi-mono/issues/1875).

## Recommended implementation boundary

- **Lowest-risk extension MVP:** `/auth-profile`; provider aliases; overlay selector; `setStatus`; migration from any extension-owned v1 store; no attempt to hook built-in `/login`. Document that users must use `/auth-profile login`, and that provider/model IDs become aliases.
- **Canonical-identity extension prototype:** wrap public OAuth providers to participate in built-in `/login`, then keep profiles in a locked `0600` sidecar and swap the selected credential into the canonical provider slot. Prototype wrapper survival across refresh/reload and define rollback/reconciliation before treating this as production-safe. It cannot transparently cover built-in API-key login.
- **Upstream-quality solution:** change `AuthStorage` and interactive auth commands as described above, then keep only selector/presentation policy in an extension if desired.
- **Do not use:** polling/watching `auth.json` to capture built-in login, monkey-patching private `InteractiveMode`/`AuthStorage` methods, or unlocked canonical-slot swapping.

## Design decisions from grilling

- Build an extension-only v1. Pi's `auth.json` is the active credential mirror;
  the extension vault is the source of truth.
- Support a tested OAuth allowlist only. OAuth credentials are the sole v1
  credential type.
- Automatically import an existing supported credential as that provider's
  `Default` profile on first run.
- Wrap the public OAuth `login()` contract to recognize built-in `/login` after
  OAuth validation and before Pi persists the active mirror. On success, create
  a new profile by default, prompt with a generated name, and make it the
  provider's default. If that prompt is cancelled, save the generated name
  automatically.
- `/auth-profile sync` explicitly reconciles a manually edited canonical
  credential: update the active profile, save a new named profile, or discard
  the change and restore the mirror.
- `/logout` detaches the canonical mirror without deleting saved profiles.
  Only the profile manager deletes profiles.
- Activation selects a provider, profile, and compatible model together.
  Defaults are stored per provider, and another Pi session may supersede the
  global active mirror (last selection wins).
- Deleting an active profile requires confirmation and switches to another
  saved profile for that provider when available; otherwise it signs out.
- While Pi runs, refresh profiles nearing expiry. Preserve failed profiles,
  mark them `Needs login`, and retry only transient failures.
- Store the global vault under `~/.pi/agent/` using a `0700` directory, `0600`
  files, a cross-process lock, atomic writes, and secret-redacted diagnostics.
  Encryption/keychain support is deferred.

## Gaps

- No official issue was found that promises native named auth profiles or defines a target schema.
- Provider-alias compatibility should be validated against every built-in provider, especially `modifyModels()` and provider-specific extra OAuth fields; source inspection establishes feasibility, not universal runtime compatibility.
- The available environment exposed file-reading/web tools but no shell, so the installed package’s test suite and git status could not be executed here.
